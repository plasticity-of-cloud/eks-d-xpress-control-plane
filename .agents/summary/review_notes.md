# Review Notes

## Consistency Check

No significant inconsistencies found. The following minor points are noted:

1. **Dual `TokenClaims` types**: `eks-dx-lambda` uses a Java `record` (`ai.codriverlabs.eksdx.lambda.model.TokenClaims`) while `eks-dx-auth-proxy` defines its own `TokenValidationService.TokenClaims` class. They are structurally similar but not shared. This is intentional (modules are independently deployable) but could cause confusion.

2. **Association GET auth discrepancy**: `sam.yaml` marks `AssociationsGet` and `AssociationGet` as `Authorizer: NONE`, while `AssociationsPost` and `AssociationDelete` use the default IAM auth. The `WebhookAuthFilter` adds an optional Bearer token check on top of the open GET endpoints. This is correct but non-obvious.

3. **CDK vs SAM divergence**: `sam.yaml` explicitly grants `iam:GetRole` on `arn:aws:iam::*:role/*`. `EksDxStack.java` only adds `sts:AssumeRole` and `sts:TagSession` — `iam:GetRole` is absent from the CDK stack. Deployments via CDK will fail at association creation unless the Lambda execution role has `iam:GetRole` from another source.

## Completeness Gaps

1. **Deployment guide**: No documentation on how to deploy `eks-dx-auth-proxy` and `eks-dx-pod-identity-webhook` to a cluster (Helm values, required RBAC, TLS certificate setup for the webhook). The `docs/user_guides/ec2-k3s-pod-identity/setup.sh` script exists but is not documented in the generated docs.

2. **Webhook TLS**: The admission webhook requires TLS. How certificates are provisioned (cert-manager, manual, Quarkus TLS) is not documented in the codebase source files reviewed.

3. **Integration test setup**: `DynamoDbIntegrationTest` uses DynamoDB Local on port 18000. The flag `-Dintegration.dynamodb=true` is mentioned in README but the exact Maven profile or property that enables it is not visible in the reviewed pom files.

4. **Native CLI build**: The `eks-dx-cli` native build requires GraalVM. The specific GraalVM version and any `reflect-config.json` or `native-image.properties` files are not documented.

5. **Webhook SA token**: The webhook reads its SA token from `/var/run/secrets/eks-dx/token`. The projected volume configuration that mounts this token is not in the reviewed source — it would be in Kubernetes deployment manifests (likely generated by Quarkus Kubernetes extension).

## Recommendations

- Add a `docs/deployment/` section covering Helm chart values for auth-proxy and webhook.
- Document the TLS setup for the admission webhook.
- Add `iam:GetRole` to `EksDxStack.java` to match the SAM template (currently missing from CDK).
- Consider extracting `TokenClaims` into a shared module to eliminate duplication.
