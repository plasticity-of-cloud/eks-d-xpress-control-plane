# EKS Auth Validation Gaps

This document outlines the differences between real EKS Auth validation and our CI/CD proxy implementation.

## Real EKS Auth Validation Flow

### 1. Token Source
- **Real EKS**: Kubernetes projects a service account token with audience `pods.eks.amazonaws.com` into the pod
- **Our Implementation**: Accepts any Kubernetes service account token

### 2. Token Validation

| Aspect | Real EKS Auth | Our Implementation | Gap |
|--------|---------------|-------------------|-----|
| **Signature Verification** | Validates against Kubernetes API server's public keys | No cryptographic verification | Security gap (acceptable for CI/CD) |
| **Audience Validation** | Requires `pods.eks.amazonaws.com` | Not validated | Missing check |
| **Token Expiry** | Validates token hasn't expired | Not validated | Missing check |
| **Issuer Validation** | Validates Kubernetes issuer | Not validated | Missing check |

### 3. Claims Extraction

| Claim | Real EKS Auth | Our Implementation | Gap |
|-------|---------------|-------------------|-----|
| `kubernetes.io/serviceaccount/namespace` | ✅ Extracted | ✅ Extracted | None |
| `kubernetes.io/serviceaccount/service-account.name` | ✅ Extracted | ✅ Extracted | None |
| `kubernetes.io/serviceaccount/service-account.uid` | ✅ Extracted | ❌ Not extracted | Missing |
| `kubernetes.io/pod/name` | ✅ Extracted | ❌ Not extracted | Missing |
| `kubernetes.io/pod/uid` | ✅ Extracted | ❌ Not extracted | Missing |

### 4. Pod Identity Association Lookup

| Aspect | Real EKS Auth | Our Implementation | Gap |
|--------|---------------|-------------------|-----|
| **Storage** | EKS API (CreatePodIdentityAssociation) | Kubernetes ConfigMap | Different storage (acceptable for CI/CD) |
| **Lookup Key** | `cluster:namespace:service-account` | `cluster:namespace:service-account` | Same |
| **Association ID** | Generated by EKS API | Generated locally | Different format |

### 5. STS AssumeRole Call

| Aspect | Real EKS Auth | Our Implementation | Gap |
|--------|---------------|-------------------|-----|
| **Principal** | `pods.eks.amazonaws.com` | Uses provided AWS credentials | Different (acceptable for CI/CD) |
| **Session Tags** | ✅ Added | ❌ Not added | Missing |
| **Transitive Tags** | ✅ Set | ❌ Not set | Missing |

### 6. Session Tags

Real EKS Auth adds these session tags:

| Tag | Real EKS Auth | Our Implementation | Gap |
|-----|---------------|-------------------|-----|
| `eks-cluster-arn` | ✅ | ❌ | Missing |
| `eks-cluster-name` | ✅ | ❌ | Missing |
| `kubernetes-namespace` | ✅ | ❌ | Missing |
| `kubernetes-service-account` | ✅ | ❌ | Missing |
| `kubernetes-pod-name` | ✅ | ❌ | Missing |
| `kubernetes-pod-uid` | ✅ | ❌ | Missing |

### 7. Response Fields

| Field | Real EKS Auth | Our Implementation | Gap |
|-------|---------------|-------------------|-----|
| `subject.namespace` | ✅ | ✅ | None |
| `subject.serviceAccount` | ✅ | ✅ | None |
| `audience` | `pods.eks.amazonaws.com` | `sts.amazonaws.com` | Incorrect value |
| `podIdentityAssociation.arn` | ✅ | ❌ | Missing |
| `podIdentityAssociation.id` | ✅ | ✅ | None |

## Implementation Plan

### Phase 1: Token Validation Enhancements
1. Add audience validation (`pods.eks.amazonaws.com`)
2. Add token expiry validation
3. Extract additional claims (pod name, pod UID, service account UID)

### Phase 2: Session Tags
1. Add session tags to STS AssumeRole call
2. Set transitive tags for cross-account scenarios

### Phase 3: Response Corrections
1. Fix audience field to `pods.eks.amazonaws.com`
2. Add pod identity association ARN to response

## Security Considerations

### Acceptable Gaps for CI/CD
- **No signature verification**: CI/CD environments are trusted; tokens come from trusted sources
- **ConfigMap-based associations**: Simpler setup for testing; no EKS API required

### Must Fix
- **Audience validation**: Prevents token reuse from other contexts
- **Token expiry**: Prevents use of stale tokens
- **Session tags**: Required for ABAC policies and proper audit trails
